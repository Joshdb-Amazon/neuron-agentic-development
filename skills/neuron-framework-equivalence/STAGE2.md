## CRITICAL: What You Are Testing

For each component, you compare the **source model's actual class** against the **target port's actual class**:

- `ref_fp32` = **Source model's class** (e.g., `Olmo2RMSNorm`, `Qwen3MLP`) in FP32
- `ref_bf16` = **Source model's class** in BF16
- `target_bf16` = **Target port's class** (e.g., `LlamaRMSNorm`, `NeuronLlamaMLP`) in BF16

**Do NOT test HF vs HF.** In CPU mode, many Neuron ports reuse HF classes (e.g., `LlamaRMSNorm` instead of `CustomRMSNorm`). If the source model uses `Olmo2RMSNorm` and the port uses `LlamaRMSNorm`, you must test `Olmo2RMSNorm` vs `LlamaRMSNorm` — these are different implementations with different precision behavior.

Read the target modeling file to find which class each component actually uses. Check `get_rmsnorm_cls()`, `__init__` methods, and imports.

**CPU mode vs Device mode:** Some ports use different classes depending on execution mode. For example, `get_rmsnorm_cls()` returns `LlamaRMSNorm` in CPU mode but `CustomRMSNorm` (NKI kernel) on device. The model tree (built in CPU mode) shows the CPU-mode class.

**Check `{EXP_DIR}/class_divergence_report.json`** (generated by Stage 0, Step 0.3) for all components with CPU/device class divergence. For each divergence, you must write TWO tests:

1. **CPU class test** (`test_NN_<component>.py`) — standard 3-tensor comparison using the CPU-mode class. This validates the CPU code path.
2. **Device algorithm test** (`test_NN_<component>_device_algo.py`) — reimplement the device class's `forward()` math in pure PyTorch and compare against the source. This validates the algorithm that will actually run on device.

Example: If `CustomRMSNorm` (device) casts to FP32, computes `x * rsqrt(mean(x^2) + eps) * weight`, casts back — write a test function that does exactly this computation and compare against the source's RMSNorm. If this test fails, the device E2E (Stage 5) will also fail even though the CPU class test passes.

**If no divergence report exists**, read the target modeling file manually. Check `get_rmsnorm_cls()`, `get_attn_cls()`, and similar factory functions. Check `__init__` methods for `if self.config.neuron_config.on_cpu:` branches.


If a leaf passes but its composite fails → bug is in composition logic.

## Writing a Test

Use [templates/conftest_template.py](templates/conftest_template.py) for scaffolding and [templates/test_template.py](templates/test_template.py) for the per-component pattern.

Key rules:
1. **Shared FP32 weights** across all three module instances
2. **`nn.Parameter()` replacement** for `ColumnParallelLinear` (not `copy_()` — it silently preserves dtype)
3. **`.eval()` mode** on Neuron modules with `pad=True`
4. **Handle tuple outputs**: `out = out[0] if isinstance(out, tuple) else out`
5. **Align shapes** before `compare_3tensors` (fused QKV, vocab padding)
6. **All outputs cast to `.float()`** before comparison
7. **Use the actual classes from both models** — not the same class for ref and target

## Running

```bash
python3 scripts/run_stage2.py --tests-dir ${EXP_DIR}/tests --tau-r 1.2 --output ${EXP_DIR}/results/stage2.json
```

The runner imports test files in-process, executes `test_*` functions, captures R-ratios from stdout.

## Interpreting Results

| R-ratio | Meaning |
|---------|---------|
| ≈ 1.0 | Port matches precision baseline. No bug. |
| 1.0 < R < 1.2 | Slight excess — may be TP rounding or kernel difference. Acceptable. |
| R >> 1.2 | Porting bug. Proceed to Stage 3. |
| R < 1.0 | Over-precision. Check for extra `.float()` calls. |
| R >> 10 | Missing algorithm (YaRN, MoE routing, etc.) |
| R >> 100 | Completely wrong computation. |
